Device Code Phishing to get access token

Posted on Jan 1, 0001
#phishing #red #ttp

As more organizations move to hardware tokens, such as Yubikeys or Windows Hello for Business attackers will look for another ways to compromise user accounts.

One such technique is by using the OAuth2 Device Code Grant Type.

Attack Flow

The attack is quite simple. The attacker starts a Device Code flow. This is one thing that ropci can do, but you can also just issue it with curl or Burp.

./ropci auth devicecode

Grab the Device Code token Microsoft returns and place it into a phishing email. The goal of the attacker is to trick the user to click the link and enter the device code. This is how such a phishing email might look like:

The goal is to trick the user to walk through the auth flow. There are a couple of steps, including entering the device code and granting the requested OAuth app (ropci by default uses Office’s client_id) to permission.

Once the victim completes the OAuth flow and grants access, the attacker receives the access token!

Attacker can now use ropci to read the users email, search mail/SharePoint, send mail, read info from all users/groups/apps/devices in the AAD tenant, generally anything that Microsoft Graph API can do. Worst case, if you an attacker phishes a powerful account might be access to other user’s data or running commands on Azure VMs, etc.

Mitigation

The easiest is likely to entirely disable the Device Code flow in your organization, unless there is a true business need to support it.

Reviewing and monitoring the device code flow initiator and the host that grants access. Both of these hosts/devices should likely be coming from a similiar IP address.

Attacker limitations

  1. There is only a short time window while the device code is valid, which the attacker has to leverage
  2. URL redirection post authentication. So far I haven’t found a way to redirect the use to another resource post authentication (this would allow an attacker to minimze being reported as suspicious)
